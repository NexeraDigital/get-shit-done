## Directory Purposes

**`bin/`:**
- Purpose: Installation orchestration
- Contains: `install.js` that detects IDE runtime and copies GSD assets to the right location
- Key files: `bin/install.js`

**`get-shit-done/`:**
- Purpose: Core GSD assets distributed to IDE config directories
- Contains: Tools library, workflow definitions, command templates, agent prompts
- Key files: `get-shit-done/bin/gsd-tools.cjs` (main CLI), `get-shit-done/bin/lib/*.cjs` (modules)

**`get-shit-done/bin/lib/`:**
- Purpose: CommonJS modules for gsd-tools functionality
- Key files:
  - `core.cjs` — Shared constants, model profiles, path helpers, output formatting
  - `config.cjs` — Config.json CRUD operations
  - `state.cjs` — STATE.md field updates, metric recording
  - `phase.cjs` — Phase CRUD (add, insert, remove, complete, list)
  - `commands.cjs` — Standalone utilities (slug generation, timestamp, todo listing)
  - `verify.cjs` — Validation commands (plan structure, phase completeness)
  - `frontmatter.cjs` — YAML frontmatter extraction and reconstruction
  - `milestone.cjs` — Milestone operations and archiving
  - `roadmap.cjs` — ROADMAP.md parsing and updates
  - `template.cjs` — Template file generation

**`get-shit-done/templates/`:**
- Purpose: Frontmatter templates for new plans, summaries, verifications
- Contains: `.md` files with `---` frontmatter blocks
- Key files: `roadmap.md`, `context.md`, `state.md`, `summary-standard.md`, `summary-minimal.md`

**`get-shit-done/references/`:**
- Purpose: Reusable prompt snippets that workflows include
- Contains: Markdown with instruction patterns, context templates
- Key files: `ui-brand.md`, context patterns for plans

**`get-shit-done/workflows/`:**
- Purpose: Orchestration instructions for interactive (IDE-driven) workflows
- Contains: Markdown files with XML blocks (objective, execution_context, process)
- Key files: `plan-phase.md`, `execute-phase.md`, `new-project.md`, `research-phase.md`

**`commands/gsd/`:**
- Purpose: Metadata declarations for IDE commands
- Contains: `.md` files with frontmatter (name, agent, allowed-tools) + optional instructions
- Key files: `plan-phase.md`, `execute-phase.md`, `research-phase.md`, `new-project.md`, `new-milestone.md`
- Pattern: Each command file is registered in IDE config (e.g., ~/.claude/prompts/) on install

**`agents/`:**
- Purpose: Specialized AI agent instructions
- Contains: XML-formatted prompts with embedded knowledge, tool lists, context injection patterns
- Key files:
  - `gsd-planner.md` — Plan creation
  - `gsd-executor.md` — Plan execution
  - `gsd-verifier.md` — Verification
  - `gsd-debugger.md` — Error analysis and fix suggestions
  - `gsd-codebase-mapper.md` — Codebase analysis (tech stack, architecture, patterns)
  - `gsd-phase-researcher.md` — Domain research
  - `gsd-plan-checker.md` — Plan validation

**`hooks/`:**
- Purpose: Git hooks and IDE integrations
- Contains: JavaScript files for pre-commit checks, status line updates, context monitoring
- Key files: `gsd-check-update.js`, `gsd-context-monitor.js`, `gsd-statusline.js`

**`autopilot/src/`:**
- Purpose: TypeScript source for autonomous orchestrator
- Key subdirectories:
  - `cli/` — Command-line interface, argument parsing, setup wizard
  - `orchestrator/` — Phase loop, gate validation, error handling
  - `claude/` — Claude Code process spawning and interaction
  - `state/` — StateStore with Zod schema validation
  - `config/` — Configuration loading with user-level defaults
  - `ipc/` — Event files, heartbeat, answer polling for dashboard-CLI IPC
  - `server/` — Express server, routes, middleware
  - `logger/`, `activity/`, `notifications/` — Supporting services

**`autopilot/dist/`:**
- Purpose: Compiled JavaScript (generated by `npm run build`)
- Contains: Output of `tsc` compiler from `src/`
- Generated: Yes
- Committed: No (git-ignored, rebuilt on npm install)

**`autopilot/dashboard/`:**
- Purpose: React web UI for dashboard
- Contains: Separate npm project with React components, Vite build
- Generated: dist/ (git-ignored)
- Built by: `npm run build:dashboard` inside dashboard/ subdirectory

**`.planning/`:**
- Purpose: Project-level state and planning artifacts
- Contains: Configuration, state file, roadmap, phase directories, todos, autopilot IPC
- Key files:
  - `.planning/config.json` — User preferences (model profile, commit strategy, verification flags)
  - `.planning/STATE.md` — Frontmatter + narrative describing project state
  - `.planning/ROADMAP.md` — Phase checklist and heading-based phase metadata
  - `.planning/phases/*/` — Phase directories with PLAN.md, SUMMARY.md, VERIFICATION.md
  - `.planning/autopilot/state.json` — Autonomous orchestrator state snapshot

## Key File Locations

**Entry Points:**
- `bin/install.js` — Installation: detects IDE, copies assets, prints success
- `get-shit-done/bin/gsd-tools.cjs` — gsd-tools CLI: state queries, phase operations, commits
- `autopilot/src/cli/index.ts` — Autopilot CLI: autonomous orchestration
- `commands/gsd/` — IDE command definitions (each file = one `/gsd:*` command)

**Configuration:**
- `.planning/config.json` — GSD project configuration
- `autopilot/package.json` — Autopilot package metadata
- Root `package.json` — Main GSD package
- `autopilot/tsconfig.json` — TypeScript compiler configuration

**Core Logic:**
- `get-shit-done/bin/lib/core.cjs` — Shared constants, model profiles, helpers
- `autopilot/src/orchestrator/index.ts` — Phase execution loop and gate logic
- `get-shit-done/bin/lib/phase.cjs` — Phase operations (CRUD, lifecycle)
- `autopilot/src/state/index.ts` — State persistence with Zod validation

**Testing:**
- `tests/` — Integration tests for main package
- `autopilot/src/**/__tests__/` — Unit tests for autopilot modules (test files colocated)
- `scripts/run-tests.cjs` — Test runner script

## Naming Conventions

**Files:**
- Markdown: `UPPERCASE.md` (PLAN.md, SUMMARY.md, ROADMAP.md, STATE.md)
- Configuration: `lowercase.json` (config.json, package.json)
- CommonJS modules: `kebab-case.cjs` (gsd-tools.cjs, core.cjs, phase.cjs)
- TypeScript: `kebab-case.ts` (orchestrator/index.ts, state/index.ts)
- Workflows: `kebab-case.md` (plan-phase.md, execute-phase.md)
- Commands: `kebab-case.md` (plan-phase.md, new-project.md)
- Agents: `gsd-kebab-case.md` (gsd-planner.md, gsd-executor.md)

**Directories:**
- Phase directories: `N/` or `N.D/` (integer or decimal phase number; e.g., `1/`, `2.1/`, `5/`)
- Feature areas: `lowercase` (autopilot/, agents/, hooks/, scripts/)
- IDE-specific: `.ide-name/` (e.g., `.claude/`, `.opencode/`, `.gemini/`, `.codex/`)

## Where to Add New Code

**New Feature (Phase-based):**
- Primary code: Write new PLAN.md in `.planning/phases/N/`
- Tests: Add test file alongside implementation or in dedicated test directory
- Planning docs: Update `.planning/ROADMAP.md` with phase description, update `.planning/STATE.md` with progress

**New Workflow/Command:**
- Workflow: Create `get-shit-done/workflows/new-workflow-name.md`
- Command metadata: Create `commands/gsd/new-workflow-name.md` with frontmatter
- Agent support: Update existing agent (or create new `agents/gsd-new-agent.md`) if needed

**New Autopilot Service (TypeScript):**
- Implementation: Create `autopilot/src/feature-area/index.ts`
- Tests: Create `autopilot/src/feature-area/__tests__/index.test.ts`
- Exports: Add to `autopilot/src/index.ts` if public API
- Build: Run `npm run build` in autopilot/ directory

**New gsd-tools Command:**
- Implementation: Add function to `get-shit-done/bin/lib/relevant-module.cjs`
- Dispatch: Update case/switch in `get-shit-done/bin/gsd-tools.cjs` to call function
- Validation: Add input validation and error handling in function

**Utilities/Helpers:**
- Shared helpers: `get-shit-done/bin/lib/core.cjs` for gsd-tools; `autopilot/src/common/` for autopilot
- IDE-specific: Create IDE-specific files in `get-shit-done/templates/` or `get-shit-done/references/`

## Special Directories

**`.planning/`:**
- Purpose: Project state root (created by gsd:new-project)
- Generated: Yes (files created by commands/workflows)
- Committed: Yes (state files and planning artifacts are committed to git)

**`.planning/phases/`:**
- Purpose: Phase work units (each subdirectory = one phase)
- Generated: Yes (created by gsd:add-phase or gsd:plan-phase)
- Committed: Yes

**`.planning/autopilot/`:**
- Purpose: Autonomous orchestrator state (only when running gsd-autopilot)
- Generated: Yes
- Committed: No (temporary runtime state, git-ignored)

**`autopilot/dist/`:**
- Purpose: Compiled TypeScript output
- Generated: Yes
- Committed: No (git-ignored, rebuilt on npm install)

**`get-shit-done/bin/lib/`:**
- Purpose: Internal gsd-tools modules
- Not intended for direct user modification; changes here affect all GSD commands
- Version-locked to main package.json version

**`hooks/dist/`:**
- Purpose: Compiled git hooks (output of build-hooks.js)
- Generated: Yes
- Committed: No (git-ignored, rebuilt on publish)
