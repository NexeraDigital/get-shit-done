<feature>
  <name>ConsolidatedEventTailer</name>
  <files>autopilot/src/ipc/consolidated-event-tailer.ts, autopilot/src/ipc/__tests__/consolidated-event-tailer.test.ts</files>
  <behavior>
    - Given an empty log directory, ConsolidatedEventTailer starts without error and emits no events
    - Given a log directory with `events.ndjson`, it tails that single file (sequential mode)
    - Given a log directory with `events-phase-1.ndjson` and `events-phase-2.ndjson`, it tails both files and emits events from each
    - Given a new `events-phase-3.ndjson` file appearing after start, it auto-discovers and begins tailing on the next tick
    - Each file has independent sequence tracking -- seq=1 in file A does not conflict with seq=1 in file B
    - getRecentEvents() returns a merged ring buffer of recent events from all files
    - stop() closes all open file handles and clears the timer
    - Events are processed in file-alphabetical order per tick (arrival order, not timestamp-sorted -- per discretion recommendation)
  </behavior>
  <implementation>
    Create ConsolidatedEventTailer extending EventEmitter with the same public API as EventTailer (start/stop/getRecentEvents/emits 'event'). Internally use a Map<string, FileTailState> where FileTailState = { handle, offset, buffer, lastSeq }. On each tick: readdir the log directory filtered to `events.ndjson` and `events-phase-*.ndjson`, open new files into the Map, read new bytes from each file using the same tail logic as EventTailer, track lastSeq per-file. Ring buffer of 200 merged events. 500ms tick interval. After tests pass, update standalone.ts to import ConsolidatedEventTailer instead of EventTailer.

    CRITICAL per Pitfall 6: scan for BOTH `events.ndjson` AND `events-phase-*.ndjson` so sequential mode still works.
    CRITICAL per Pitfall 1: track lastSeq per file, NOT globally.
    File handle strategy per Pitfall 2: keep handles open (cheap), close all on stop().
  </implementation>
</feature>
